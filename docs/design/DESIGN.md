# Design System — Cai

Generated by /design-consultation on 2026-04-01. Updated 2026-05-02 with reusable patterns surfaced during the action-list-polish work (PR #22 + #23 follow-ups).

Update this file when any visual or UI decision changes.

---

## Product Context

- **What this is:** Native macOS menu bar clipboard manager powered by local AI
- **Who it's for:** Developers who live in 5+ apps daily and want zero context-switching
- **Space:** macOS productivity tools (peers: Raycast, Alfred, Spotlight, Paste)
- **Project type:** Native macOS app (SwiftUI + AppKit), fixed-width popover (~380pt)
- **Tone:** Invisible, precise, trustworthy — a tool that earns its place in muscle memory

---

## Aesthetic Direction

- **Direction:** Precision Minimal (Spotlight lineage)
- **Decoration level:** Minimal — frosted glass does all the work. No gradients, no shadows as decoration, no patterns.
- **Mood:** The app should feel like it *belongs* to macOS — not a wrapper, not a web app. Every pixel is justified. Nothing competes for attention except the action the user is about to take.
- **References:** Apple Spotlight, Linear, Raycast (structural discipline), Vercel (one accent does everything)

### Research findings (2026-04-01)

| Peer | Accent | Font | Notes |
|------|--------|------|-------|
| Raycast | Coral #FF6363 | Inter | Dark-first, glass-morphism |
| Linear | Indigo ~#5E6AD2 | Inter | Tight density, zero decoration |
| Vercel | Blue #0070F3 | Geist Sans | Swiss minimalism, monochrome-dominant |
| Zed | Electric blue ~#3355FF | SF Pro/IBM Plex | Utilitarian-beautiful, speed as aesthetic |
| Spotlight | System blue (NSColor) | SF Pro | `.hudWindow` blur, adaptive NSColors, 36-44pt rows |

**Positioning:** Cai's indigo (#6366F2) sits between Linear (calm authority) and Raycast (energy). It signals developer taste without mimicking either directly.

## Color

**Approach:** Restrained — one accent, all surfaces adaptive.

### Custom tokens (in `CaiColors.swift`)

```swift
extension Color {
    // Accent
    static let caiPrimary       = Color(red: 0.39, green: 0.40, blue: 0.95)  // #6366F2 — indigo-500
    static let caiPrimarySubtle = Color(red: 0.39, green: 0.40, blue: 0.95).opacity(0.12)  // hover/selection wash

    // Semantic
    static let caiSuccess       = Color(red: 0.204, green: 0.780, blue: 0.349)  // #34C759 — Apple system green
    static let caiError         = Color(red: 1.0, green: 0.584, blue: 0.0)      // #FF9500 — Apple system orange

    // Surfaces & text — adaptive via NSColor (auto dark/light, never hardcode)
    static let caiBackground    = Color(nsColor: .windowBackgroundColor).opacity(0.95)
    static let caiSurface       = Color(nsColor: .controlBackgroundColor)
    static let caiTextPrimary   = Color(nsColor: .labelColor)
    static let caiTextSecondary = Color(nsColor: .secondaryLabelColor)
    static let caiSelection     = Color(nsColor: .selectedContentBackgroundColor).opacity(0.15)
    static let caiDivider       = Color(nsColor: .separatorColor)
}
```

### Design rules
- **Never hardcode a background color.** `caiBackground`, `caiSurface`, and all text colors are NSColor-based — they auto-adapt to Light / Dark / System modes.
- **`caiPrimary` is the only hardcoded color** — it's the brand. It reads well on both dark (#1C1C1E) and light (#F5F5F7) surfaces (≥4.5:1 contrast ratio on white).
- **`caiPrimarySubtle`** replaces `caiSelection` for indigo-branded interactive states (row hover, in-progress step background wash). `caiSelection` (system blue) is kept for text selection only.
- **Success = green, Error = orange.** These match Apple's semantic system colors (SF Symbol `checkmark.circle.fill`, `exclamationmark.circle.fill`). Users already know what they mean.

### Indigo discipline

`caiPrimary` (indigo) signals "I will act / I am on / I am the focus." It's the only chromatic moment in the app — overusing it dilutes the signal. Reserve it for:

- Primary buttons (Save, `+`, "Run")
- ON-state toggles (`ChipToggle` filled — Silent, Auto-replace, Show-in-action-list)
- Focused input border (`MultilineTextEditor`, `ChainStepsTokenField`)
- Pinned-state pin glyph + tinted background
- Header product mark (the icon on every screen header)
- Dropdown / list-row keyboard-or-mouse selection highlight

**Do NOT use `caiPrimary` for passive structural elements:**

- Tab indicators — neutral `windowBackgroundColor` pill on neutral track (see `TabBar.swift`); it's structure, not action
- Individual chain-step chips (`ChainStepsTokenField`'s `.action` / `.inlineLLM` chips) — they're labels ("I am a step"), not actors. Use `caiSurface.opacity(0.6)` fill, `caiDivider.opacity(0.5)` border, `caiTextSecondary` icon.
- List-row hover backgrounds — use `caiSurface.opacity(0.4)`. Indigo here would compete with the row's inner indigo elements (pin, save).
- Section headers, helper text, dividers, secondary buttons.

The mental test: if a screenshot of this element existed in isolation, would the user expect tapping it to *do* something? Indigo means yes. Anything that's just *describing* state belongs in the neutral palette.

### Dark mode strategy
Three modes: **System** (default), **Light**, **Dark** — implemented via `NSApp.appearance` in `CaiSettings`. No additional color work needed — all NSColor tokens adapt automatically.

## Typography

**Font:** SF Pro — the macOS system font. Zero setup, perfect native feel, identical to Spotlight, Terminal, and every first-party Apple app.

- `SF Pro Text` is used at ≤19pt (all Cai UI sizes)
- `SF Pro Display` would be used at ≥20pt — not currently needed in the app

### Size scale

| Role | Size | Weight | Usage |
|------|------|--------|-------|
| Header title | 13pt | Semibold | Screen titles (ResultView header, progress view header) |
| Body / action label | 13pt | Medium | ActionRow labels, step labels, form field values |
| Body regular | 13pt | Regular | LLM output text, result content, long-form |
| Control row label | 12pt | Medium | Toggle row labels in Settings sub-screens (`ConnectorsSettingsView`, `BuiltInActionsView`). One step down from body so the toggle reads as the primary affordance, not the label. |
| Secondary | 11pt | Regular | Subtitles, result summaries, step result snippets |
| Footer hints | 10pt | Medium | KeyboardHint chips |
| Micro / badges | 9pt | Medium | `⌘1`–`⌘9` shortcut badges |

### Typography rules

**Rule 1: `.rounded` design variant for accent numerals only.**
Use `Font.system(size: X, design: .rounded)` for:
- Keyboard shortcut badges (`⌘1`–`⌘9`)
- Step count indicators
- Standalone result counts ("3 found", "2 duplicates")

All labels, body text, and UI copy use the default SF Pro (no `design:` modifier). The rounded variant adds warmth exactly where the app most needs to feel helpful — at decision points and result moments.

**Rule 2: `.monospacedDigit()` wherever numbers change.**
Apply to all numerical content that updates at runtime: step counts, shortcut numbers, result counts. Prevents micro layout-jitter as digits change width (1→2→3).

```swift
// Example
Text("\(stepCount) found")
    .font(.system(size: 11, design: .rounded))
    .monospacedDigit()
```

**Rule 3: Never use Dynamic Type.** macOS doesn't support it. All sizes are fixed pt values.

## Window & Background

**Material:** `NSVisualEffectView` with `.behindWindow` blending mode. Light mode uses `.hudWindow` (frosted glass, same as Apple Spotlight). Dark mode uses `.underWindowBackground` (properly dark — `.hudWindow` is too translucent in dark mode). `window.backgroundColor = .clear`, `isOpaque = false`.

**Indigo tint (Risk #1):** Apply a faint indigo overlay to `VisualEffectBackground.swift`:
```swift
// A barely-there indigo cast — 4-5% opacity
// Makes the window feel like Cai before the user reads a word
Color.caiPrimary.opacity(0.05)
    .allowsHitTesting(false)
```

**Window dimensions:** 540pt wide (fixed). Height varies by screen (ScrollView with per-section maxHeight constraints). Corner radius: 20pt. The wider width accommodates MCP forms, subtitles, and destination chips comfortably — closer to Raycast than Spotlight.

**Base unit:** 4px

| Token | Value | Usage |
|-------|-------|-------|
| `spacing2xs` | 4pt | Icon inner padding, tight gaps |
| `spacingXs` | 6pt | Row internal gaps (icon↔text) |
| `spacingSm` | 8pt | Small internal padding |
| `spacingMd` | 12pt | Row horizontal padding (confirmed in ActionRow) |
| `spacingLg` | 16pt | Section horizontal padding (header, form, progress view) |
| `spacingXl` | 24pt | Between major sections |
| `spacing2xl` | 32pt | Large separations |
| `spacing3xl` | 48pt | Max content separation |

**Row heights:**
- Standard action row: 42pt (label only), 56pt (label + subtitle)
- Progress step row: 44pt (label only), 60pt (label + result)
- Header row: 38pt (icon + title)
- Footer row: 32pt (keyboard hints)

## Border Radius

| Role | Value | Elements |
|------|-------|---------|
| Micro | 4pt | Keyboard shortcut chips, small badges |
| Small | 6pt | Icon containers (28×28) |
| Medium | 8pt | Action rows (selection background), cards, form inputs |
| Large | 10pt | Panels, popovers within the window, settings sub-screen toggle row cards |
| Window | 20pt | The Cai popover window itself |

**Rule:** Radius scales with element size. A 28px icon container at 6pt feels proportionally correct. A full-window border at 4pt looks underbaked.

## Motion

**Approach:** Minimal-functional. Every animation aids comprehension. None are decorative.

| Name | Curve | Duration | Usage |
|------|-------|----------|-------|
| Transition | `.easeOut` | 0.2s | Screen changes, step completions, result reveals, error state appearance |
| Micro | `.easeOut` | 0.15s | Row selection highlight, button press feedback |
| Pulse | `.easeInOut` repeating | 1.2s | In-progress ◉ step indicator (opacity 0.45↔1.0) |

**The rule:** No bounce. No spring physics on utility interactions. Bounce signals "consumer app." Utility tools that interrupt your workflow should move with intent, not personality.

```swift
// Standard transition — use this everywhere
withAnimation(.easeOut(duration: 0.2)) { ... }

// Pulse — in-progress step indicator only
.opacity(isPulsing ? 1.0 : 0.45)
.animation(.easeInOut(duration: 1.2).repeatForever(autoreverses: true), value: isPulsing)
```

## Reusable Component Patterns

### Settings sub-screen toggle row

Established by `ConnectorsSettingsView`, reused by `BuiltInActionsView`. Use this layout for any future Settings sub-screen that boils down to "list of items, each with on/off."

```
┌────────────────────────────────────────────────────────┐
│  [icon]  Label                                [Toggle] │
│          Subtitle (scope, status, or hint)             │
└────────────────────────────────────────────────────────┘
```

- **Container:** `RoundedRectangle(cornerRadius: 10).fill(Color.caiSurface.opacity(0.3))`. Padding: `.horizontal(12)`, `.vertical(10)`.
- **Leading icon:** SF Symbol, `font(.system(size: 14, weight: .medium))`, in a `frame(width: 20)` to align rows. `caiPrimary` when the row's state is "on/active" / `caiTextSecondary.opacity(0.5)` when off. Visual cue for state at a glance.
- **Label:** 12pt medium, `caiTextPrimary` (Control row label scale).
- **Subtitle:** 10pt regular, `caiTextSecondary.opacity(0.7)`. Carries scope/status/hint info — not decoration.
- **Toggle:** `.toggleStyle(.switch).controlSize(.mini).tint(.caiPrimary).labelsHidden()`. Right-aligned via `Spacer()`.
- **No chevron** unless the row drills into a sub-detail (Connectors expands inline; Built-in Actions doesn't, so no chevron).

### Pin button (progressive disclosure)

Established by `ClipboardHistoryView`, reused by `ShortcutsManagementView`. Use this when an item has a binary "elevated/normal" state and the row already has a leading icon.

- **Pinned (always visible):** `pin.fill` at 12pt medium, `caiPrimary`, on a `caiPrimary.opacity(0.15)` rounded background.
- **Unpinned + hovered:** `pin` (unfilled) at 12pt medium, `caiTextSecondary.opacity(0.5)`, on `caiSurface.opacity(0.6)`. Click to pin.
- **Unpinned + not hovered:** show the row's regular icon (e.g., the shortcut type icon). Hides the pin affordance until needed.
- **Helper text:** `.help(item.isPinned ? "Unpin" : "Pin to top")` — both states have a tooltip.

The icon-doubles-as-button keeps the row visually quiet for the 90% of items the user doesn't need to elevate.

### Drag-to-reorder lists

Used in `ShortcutsManagementView` to maintain a "pinned-first, then unpinned" invariant via post-drop sort.

- Wrap the rows in a SwiftUI `List` (not `ScrollView { VStack }`) so `.onMove(perform:)` works.
- Strip List chrome to keep the existing card aesthetic: `.listStyle(.plain)`, `.scrollContentBackground(.hidden)`, and per-row `.listRowSeparator(.hidden)` + `.listRowBackground(Color.clear)` + tight `.listRowInsets`.
- Enforce invariants in the move handler, not via section boundaries: take the source/destination indices, mutate, then re-sort the underlying array. Cross-section drags snap back on drop, which feels intentional with SwiftUI's row animation.

```swift
private func moveShortcut(from source: IndexSet, to destination: Int) {
    var working = orderedShortcuts
    working.move(fromOffsets: source, toOffset: destination)
    settings.shortcuts = working.filter(\.pinned) + working.filter { !$0.pinned }
}
```

The same pattern is used in `DestinationsManagementView` for Custom destinations — `moveCustomDestination(from:to:)` operates on the pinned-first `customDestinations` view, then writes built-ins + resorted custom slice back to `settings.outputDestinations`. Built-ins live in a separate tab and stay in fixed order.

### Two-tab management screen

The shared parent shell behind `ActionsManagementView` and `DestinationsManagementView` (and any future Settings sub-screen with the same shape: a Custom tab + a Built-in tab). Lives in `ManagementScreen.swift`.

**Anatomy:**

```
┌────────────────────────────────────────────────────────┐
│  [icon]  Title                                     [+] │   ← header
│          Per-tab subtitle                              │
├────────────────────────────────────────────────────────┤
│  [ Custom (12) ]  [ Built-in (8) ]                     │   ← TabBar (neutral)
├────────────────────────────────────────────────────────┤
│                                                        │
│  Tab content (caller-supplied @ViewBuilder)            │   ← typically a
│                                                        │     chrome-stripped
│                                                        │     `List` so the
│                                                        │     Custom tab can
│                                                        │     drag-to-reorder
│                                                        │
├────────────────────────────────────────────────────────┤
│  [Esc] Back                                            │   ← footer
└────────────────────────────────────────────────────────┘
```

- **Header:** `icon · title · subtitle` on the left (icon at 14pt medium `caiPrimary`, title 13pt semibold `caiTextPrimary`, subtitle 11pt `caiTextSecondary`). The `+` on the right is rendered ONLY when `selection == customTabId` — Built-in tabs can't accept user-created items.
- **TabBar:** neutral active pill on a recessed neutral track (no indigo — see "Indigo discipline"). Custom always first, Built-in always second; matches the user's likely edit frequency.
- **Tab content:** caller-supplied `@ViewBuilder` that switches on `selection`. The child view should render WITHOUT its own header / footer — the shell provides those. When a tab needs drag-to-reorder, it uses `List` directly inside this slot, with `.listRowSeparator(.hidden) / .listRowBackground(.clear) / tight .listRowInsets` per row to strip List chrome.
- **Footer:** single `KeyboardHint(key: "Esc", label: "Back")`. Add other hints via the caller if needed.
- **`+` button trigger:** the parent's `onAdd` callback should bump a UUID `@State` that the embedded child view watches via an `externalAddTrigger` parameter. The child's `.onChange` then calls its internal `cancelForm() + beginAdding()` flow — same code path as the child's own `+` button. No `.id()` re-mount needed.

This shell guarantees both screens are pixel-identical at the chrome level and lets new tabbed Settings sub-screens drop in with ~10 lines of glue.

### Composer text input

Multi-line form fields use `MultilineTextEditor` (an `NSTextView` wrapper — see file for the rationale). Defaults:

- **`minLines: 1, maxLines: 4`, scroll beyond.** Matches Linear / Notion / Apple Mail composer UX. Two-line defaults are a web-form holdover; modern macOS composers all start at one line and grow with content.
- **Font:** 12pt system. Pass `monospaced: true` for code/template content (URLs, shell commands, JSON, AppleScript).
- **`Return` inserts a newline.** `⌘⏎` fires the form's save callback; `Esc` fires cancel. Both bypass SwiftUI's `keyboardShortcut` chain (which doesn't cooperate with `NSTextView` reliably).
- **Visual shell:** wrap with the `formFieldShell(focused:)` modifier. Background is `Color(nsColor: .textBackgroundColor)`; border is hairline `caiDivider.opacity(0.5)`, or 1px `caiPrimary.opacity(0.5)` when focused. The focus ring is the only indigo moment on this control.
- **Sizing:** clamp the field's height with `editor.minHeight ... editor.maxHeight` (computed from `fontSize × 1.4 × line-count`).

```swift
let editor = MultilineTextEditor(
    text: $value,
    placeholder: "What should this action do?",
    onCommit: { saveForm() },
    onCancel: { cancelForm() }
)
editor
    .frame(minHeight: editor.minHeight, maxHeight: editor.maxHeight)
    .formFieldShell()
```

### Empty states

Used inside management screens when a tab has zero items. Implemented as `ManagementEmptyState` (see `ManagementScreen.swift`).

**Anatomy:** icon + title + description + optional CTA button.

- **Icon:** SF Symbol, 28pt light, `caiTextSecondary.opacity(0.4)`. Pick a symbol that depicts the *kind* of thing missing (`bolt.circle` for Actions, `paperplane.circle` for Destinations).
- **Title:** 13pt medium, `caiTextPrimary`. Short noun phrase ("No custom destinations yet").
- **Description:** 11pt, `caiTextSecondary.opacity(0.7)`, centered, max ~50 chars per line. Tells the user what the thing IS in plain language ("Send results to webhooks, AppleScript, deeplinks, or shell commands.").
- **Optional CTA:** neutral button matching the `ChipButton` vocabulary — `caiSurface.opacity(0.6)` fill, hairline divider border. Used to expose the most likely next step (usually "Browse Community Extensions").
- **Layout:** centered vertically inside `minHeight: 120`, `padding(.vertical, 24)`. The empty state should feel like a soft pause, not a dead screen.

The mental rule: an empty state is a *welcome*, not an error. Warmth + one clear next step.

## Accessibility

- **Minimum touch target:** 44×44pt for all interactive elements
- **Step indicators:** always have `.accessibilityLabel("Completed")`, `"In progress"`, `"Pending"`, `"Failed"` — never rely on the visual symbol alone
- **Pulsing ◉:** add `.accessibilityAddTraits(.updatesFrequently)` to suppress VoiceOver chatter on animation frames
- **State change announcements:** post `UIAccessibility.post(notification: .announcement, argument: "Step N complete: [label]")` on each step transition
- **Keyboard navigation:** all interactive elements reachable via Tab; `⌘1`–`⌘9` shortcuts are the primary interaction model — they must work before mouse does
- **Contrast:** `caiPrimary` (#6366F2) on white = 4.52:1 (WCAG AA). On dark (#1C1C1E) = 6.1:1. Both pass.
- **Hover-only affordances:** anything that appears on hover (e.g., the unpinned pin button) must also be reachable via the form/edit flow so keyboard-only users can perform the action. `ShortcutsManagementView`'s pin toggle in the edit form covers this.
